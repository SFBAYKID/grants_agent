Adversarial review 2026-07-19 of the wired `sources/rfp_aggregator.py` (Starbridge listing) and its interaction with `scoring.py`, `slack/search.py`, `slack/drip.py`, `db.rfp_candidates`, and the LinkedIn->Salesforce contact-record path. The aggregator is the ACTUAL wired RFP source (env `RFP_DISCOVERY_ENABLED=1`); `sources/rfp.py` (the strict host+adjacency single-page flow) is NOT wired. So the loose aggregator, not the strict parser, is what ships.

**Why this matters:** the aggregator trusts a third-party listing verbatim and stamps every row `VerificationStatus.VERIFIED`, yet it does none of the primary-source cross-checks (`entity_matches_host`, `label_adjacent_date`) that the single-page flow does. Its facts reach the drip and search under the same VERIFIED gate as host-proven awards.

**Durable fragilities to re-check on any aggregator/RFP work:**

1. **State from a bare full-name substring is both a misfile (honesty) and a recall sink.** `_row_state` returns the first target-state NAME found ANYWHERE in the row block — prose, buyer name, AND url slugs (word-boundary matches inside hyphenated slugs like `/buyer/california-...`). Two failure modes, both proven on the committed fixture:
   - **Misfile / wrong state (blocker):** dict-iteration order checks `washington` first, so "Washington County, **Pennsylvania**" -> `WA` even though PA (a target) is named right there. Any City/County/Township/School named after a target state ("City of California, MO"; "Texas Township, MI"; "Mount Washington SD, KY") is stamped the wrong state, labeled VERIFIED. Violates the module's own "no guessed state" docstring and Constitution rule 1.
   - **Recall loss (serious):** the real fixture names the CITY + 2-letter code, not the spelled state, so genuine TARGET rows drop silently — City of Lockhart **TX** (dropped, no "Texas"), Lakeside Union School District **CA** (a school — the bullseye — dropped, no "California"), Eden Christian Academy **PA** (dropped). Poller returns a short list and cannot tell "no TX RFPs" from "TX RFPs whose row omitted the word Texas." Coverage-honesty gap.
   - Fix direction: parse a positional trailing ", <State>"/2-letter code near the buyer, or cross-check the detail page host; never a free substring over the whole block incl. slugs.

2. **Content-based `rfp_item_id` over-merges distinct solicitations.** Key = `entity | first-6-title-tokens | due_iso`. On the fixture, the two DISTINCT PA Pine Grove packages ("...General and HVAC Construction" vs "...Plumbing Construction *REBID*", same buyer, same Close date, same first 6 tokens) collapse to ONE lead — the second is silently dropped. `test_rfp_aggregator.py` MISLABELS this as "the same PA RFP listed twice collapses to one" and asserts unique ids as correct, so the suite HIDES the over-merge. Multi-package / phased solicitations are common; the key needs the rfp_number or a longer/distinguishing token, not a 6-word prefix.

3. **Frozen-grade staleness: a closed RFP reads as "open solicitation" in search (serious honesty gap, cross-cutting all RFP sources).** `lead_grade` is computed at ingest and only re-graded when the row is re-ingested. Once an RFP's deadline passes it drops out of the poll feed (parse drops `due < today`, Starbridge delists), so it is NEVER re-graded — `lead_grade` stays `silver`/`gold`, `funds_end` stays in the past, `status` stays `new`. `db.rfp_candidates` and `search_leads(open_only=True)` both gate on `date(funds_end) >= date('now')`, so DRIP is protected — BUT plain `search_leads(grade='silver')` / `record_kind='solicitation'` has no such gate and `search.py`'s static grade_phrases label silver as "silver (**open** solicitation)". A rep is told a week-closed RFP is an open solicitation. `open_only` is opt-in and off by default. Fix: a daily sweep re-grading past-due RFP leads to WATCH (or status='expired'), OR derive open/closed from `funds_end` vs today at presentation time instead of trusting the frozen grade.

4. **Aggregator `VERIFIED` label is overloaded.** Every aggregator row is `VerificationStatus.VERIFIED` on Starbridge's word alone (no host bind, no adjacency). `db.rfp_candidates` filters `verification_status='verified'`, and `build_rfp_alert` then asserts as fact "<entity> has an open RFP for <subject>, responses due <date>." If Starbridge mis-attributes a buyer or lists a cancelled/stale RFP, Grant asserts it confidently. Mitigations that DO hold: the drip carries the Starbridge detail link, and build_rfp_alert never claims a posting date (so a mis-GOLD only reorders, never surfaces a false "just posted"). Recommend a distinct verification tier or softened phrasing ("Starbridge lists an open RFP…") for aggregator-sourced rows.

5. **LinkedIn person name is never validated before becoming a Salesforce person Lead.** `finder.linkedin_person` takes `parts[0]` of a " - "-split search snippet as the name with zero check that it's a person (not a role). A snippet led by a title ("IT Director - City of Kemah | LinkedIn") yields name="IT Director" -> `split_person_name` -> FirstName="IT", LastName="Director" -> a fabricated-person Lead (disclosed only as "ownership not verified," which doesn't catch title-as-name). Human preview is the only gate. Plausible, not yet proven on a real-LinkedIn sample. Fix: reject `parts[0]` that matches a role/title lexicon or isn't 2+ name-shaped tokens.

6. **Parser drift on the listing is fail-closed but silent.** `_CLOSE_RE`/`_RELEASE_RE` allow only `[A-Za-z0-9,/\s]`; a Close value carrying a time or a "-" (e.g. "Jul 30, 2026 - 2:00 PM") fails to match -> row dropped. If Starbridge renders dates inline/with times, ALL rows drop and `poll()` returns `[]` (page is >300 chars, so no SourceUnreachable) -> reads as "no open RFPs." Only literal "Unavailable" within 20 chars after the title is treated as closed; a future-dated "Closed"/"Awarded"/"Cancelled" row would pass as open.
